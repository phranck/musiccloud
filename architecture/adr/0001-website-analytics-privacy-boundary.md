# ADR 0001: Website Analytics Privacy Boundary

## Status

Accepted.

## Context

The public website has no login and must not invent a real "user" identity from analytics signals. The analytics model
therefore needs three separate levels:

- `network_cluster`: a pseudonymous estimate for a connection or household.
- `device`: a browser/device estimate, preferably from a first-party visitor identifier when the chosen privacy and
  consent model allows it.
- `session`: a short-lived visit flow generated by the website.

The dashboard must expose these as estimates. It must not label them as real users or proven households.

## Decision

Website analytics stores curated product events in Postgres and keeps raw request identifiers outside persistence.

Raw IP addresses are only allowed inside request-scoped derivation code. The persisted cluster key must be a server-side
HMAC over a truncated IP prefix and a rotating period. This is pseudonymisation, not anonymisation. A plain SHA hash of
an IP address is forbidden because the IP search space is small enough for practical precomputation.

The implementation must not add browser fingerprinting signals. Canvas, font, plugin, audio and hardware probes are
explicitly out of scope. Coarse operational fields such as route template, device class, browser family, OS family,
viewport bucket and platform are allowed when they are stored as curated categories and not as a stable fingerprint.

Heatmap events may store relative coordinates and semantic element keys only. They must not store form input, selected
text, DOM snapshots, exact CSS selector paths, screenshots, continuous mouse movement or admin-page interactions.

## Data Boundaries

Allowed event families:

- `page_view`
- `search_submitted`
- `resolve_started`
- `resolve_succeeded`
- `resolve_failed`
- `listen_on_clicked`
- `similar_artist_clicked`
- `popular_track_clicked`
- `upcoming_event_clicked`
- `player_started`
- `player_paused`
- `player_resumed`
- `player_completed`
- `player_unavailable`
- `info_page_clicked`
- `help_page_clicked`
- `ui_click`

Allowed identity fields:

- `session_id`: random UUID.
- `device_key`: optional pseudonymous first-party key.
- `network_cluster_key`: pseudonymous HMAC-derived network cluster key.
- `confidence`: `low`, `medium` or `high`, describing how strong the cluster estimate is.

## Consequences

The backend schema uses dedicated website analytics tables rather than extending Umami or Apple client telemetry.
Dashboard copy and API responses must use terms such as "network cluster", "device", "session", "estimated" and
"pseudonymous". They must not claim anonymous unique users.

Any future code that derives `network_cluster_key` must carry a local privacy-boundary comment explaining that raw IPs
are not persisted, HMAC must not be replaced by plain hashing, and browser fingerprinting signals must not be added.
